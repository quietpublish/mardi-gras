## Findings

### 1. Medium: Gas Town can be opened through some paths without the data-fetch/bootstrap work used by `ctrl+g`

Status: Resolved via `mg-452`

Evidence:
- `ctrl+g` initializes the panel with convoy, mail, cost, activity, vitals, and status fetches in `internal/app/app.go:1339-1355`.
- The command-palette path only flips `showGasTown`, sets status, and maybe starts the tick in `internal/app/app.go:1928-1940`.
- The layout-cycle path does the same in `internal/app/app.go:1945-1967`.
- The view renderer prioritizes Problems over Gas Town in `internal/app/app.go:2463-2467`.

Impact:
- Opening Gas Town from the palette or via layout cycling can show a partially initialized panel.
- If Problems is already visible, the palette toggle does not clear `showProblems`, so the Gas Town panel can remain hidden even though the action succeeded internally.

Recommendation:
- Route all Gas Town activation paths through one helper that performs the same initialization as `ctrl+g`, including `showProblems = false`, data fetches, and an initial gated status poll when needed.

Implemented:
- Gas Town activation now routes through one bootstrap path across keybinding, palette, and layout-cycle entry points.
- Entering Gas Town also clears Problems state consistently before rendering the panel.

### 2. Medium: The detail pane can show stale data when the parade no longer has a selection

Status: Resolved via `mg-452`

Evidence:
- `syncSelection()` only updates the detail pane when `m.parade.SelectedIssue != nil` in `internal/app/app.go:2113-2118`.
- `rebuildParade()` always calls `m.syncSelection()` after rebuilding the left pane in `internal/app/app.go:2222-2274`.

Impact:
- If filtering or focus mode reduces the parade to zero selectable issues, the right pane keeps rendering the previously selected issue instead of clearing to an empty state.
- This creates an inconsistent UI: the left pane can say "No issues found" while the right pane still shows stale issue data.

Recommendation:
- Make `syncSelection()` clear the detail panel when no issue is selected, or explicitly clear detail state after a parade rebuild with no selectable items.

Implemented:
- Detail state now clears when filtering or focus changes leave the parade with no selection.

### 3. Medium: Git branch creation uses the process working directory instead of the resolved project directory

Status: Resolved via `mg-452`

Evidence:
- `createAndSwitchBranch()` shells out with `git checkout -b <branch>` in `internal/app/app.go:1816-1833`.
- Unlike agent launch code, it never sets `Cmd.Dir` to `m.projectDir`.

Impact:
- When `mg` is pointed at another workspace via `--path`, launched from a subdirectory, or used in redirected Beads setups, branch creation may run in the wrong repository or fail unexpectedly.

Recommendation:
- Set `cmd.Dir = m.projectDir` before running `git checkout -b`.

Implemented:
- Branch creation now runs in `m.projectDir`, matching the rest of the project-aware git and agent-launch behavior.

### 4. Low: Parade rebuilds do duplicate full-dataset work on hot paths

Status: Resolved via `mg-452`

Evidence:
- On every file-change message, the app groups the full issue set in `internal/app/app.go:713-718`.
- `rebuildParade()` may regroup filtered issues in `internal/app/app.go:2239-2246`, then always creates a new parade in `internal/app/app.go:2259`.
- `views.NewParade()` groups the issue set again and rebuilds the issue map in `internal/views/parade.go:65-89`.

Impact:
- The hot path for reloads and filter updates performs repeated `GroupByParade` and `BuildIssueMap` work.
- Because grouping also evaluates dependencies, this scales worse than a simple slice copy and is likely to become noticeable on the "closed-heavy" large datasets this project explicitly targets.

Recommendation:
- Pass precomputed groups/maps into the parade model, or cache them at the app level and reuse them across rebuilds.

Implemented:
- Parade rebuilds now reuse precomputed grouping data instead of rebuilding group/maps redundantly on hot paths.

### 5. Low: Activity loading is O(file size) even though the UI only displays the last few events

Status: Resolved via `mg-452`

Evidence:
- `fetchActivity()` always calls `LoadRecentEvents(path, 20)` in `internal/app/app.go:2421-2424`.
- `LoadRecentEvents()` uses a fixed-size ring buffer, but still scans the entire `.events.jsonl` file from the beginning in `internal/gastown/activity.go:35-85`.

Impact:
- Memory use stays bounded, but latency grows with the event log.
- Large long-lived Gas Town installs will pay full-file scan cost to render only the newest 20 events.

Recommendation:
- Consider reverse-tail style reading, persisted file offsets, or caching the last file size/modtime and only parsing the appended suffix.

Implemented:
- Activity loading now caches parsed state and appends incrementally instead of rescanning the full event log on each refresh.
